# Pachner Invariant Survival Protocol

Status: Repository-governance protocol

## Core Claim

`pachner-invariant` is presented as an executable Lean-facing artifact for local Pachner-move invariant verification, frontier tracking, and status-normalized combinatorial topology claims.

It does not claim that every broader triangulation, manifold, or Pachner-theoretic theorem is solved.

It requires every Pachner-facing claim to distinguish:

- solved theorem,
- closed executable surface,
- certified frontier,
- conditional result,
- open obstruction.

## Survival Rule

A Pachner artifact survives skeptical review when a reader can determine:

1. the target theorem or construction,
2. the verified Lean or executable surface,
3. the active mathematical frontier,
4. the command that checks the status,
5. whether any theorem-level ingredient remains open.

## Forbidden Claims

`pachner-invariant` must not imply that an open theorem is solved because:

- tests pass,
- Lean builds,
- documentation is complete,
- a constructor exists,
- a local case is verified,
- a frontier has been isolated,
- an invariant candidate has been recorded.

## Normal Form

A Pachner component is admissible only when it is one of:

1. a verified Lean theorem or executable certificate,
2. a status-normalization artifact,
3. a certified frontier statement,
4. a conditional result with explicit assumptions,
5. an open obstruction with a named missing ingredient.

## Canonical Public Claim

The durable contribution of `pachner-invariant` is the conversion of local Pachner-move and invariant claims into auditable, Lean-facing, status-normalized artifacts.

## Continuation Rule

Continue `pachner-invariant` only by:

- strengthening Lean verification,
- clarifying theorem status,
- shrinking a frontier,
- proving a registered lemma,
- improving auditability,
- removing ambiguity.

Do not expand Pachner merely by adding new terminology.
